以下为个人翻译部分：

React和React Native仅仅是框架，并没有规定我们应该如何构建我们的项目。这一切都取决于您的个人品味和您正在进行的项目。

在这篇文章中，我们将介绍如何构建项目以及如何管理本地资产。这当然不是一成不变的，你可以自由地选择适合您的部分。希望能对您有所帮助。

对于一个通过react-native init创建的项目，我们只得到基本的结构。

包括Xcode项目的ios文件夹和Android项目的android文件夹，以及作为React Native入口点的index.js文件和App.js文件。

在Windows Phone，iOS和Android上使用native，我发现构建项目都归结为按类型或功能组织文件。

type vs feature

按类型分隔意味着我们按类型组织文件。如果它是一个组件，则有容器"container"和表示文件。如果是Redux，则有action，reducer和store文件。如果是view，则有JavaScript，HTML和CSS文件。

Group by type

redux
  actions
  store
  reducers
components
  container
  presentational
view
  javascript
  html
  css

这样，我们可以看到每个文件的类型，并轻松地向某个文件类型运行脚本。这对所有项目来说都是通用的，但它没有回答“这个项目是关于什么的?“问题。这是新闻App吗？它是一个忠诚度App吗？它是关于营养追踪的吗？

按类型组织文件适用于机器，而不适用于人。很多时候我们处理一个功能，在多个目录中找到要修复的文件是一件麻烦事。如果我们计划从项目中创建一个框架，那也是一种痛苦，因为文件遍布很多地方。

Group by feature

更合理的解决方案是按功能组织文件。与某项功能相关的文件应放在一起。测试文件应贴近源文件。查看此文章以了解更多信息。

功能可以与登录，注册，用户引导或用户的个人资料相关。功能可以包含子功能，只要它们属于同一个流程。如果我们想要移动子功能，那将很容易，因为所有相关文件已经组合在一起。

除了传统的App.js文件，index.js文件，ios文件夹和android文件夹，我把所有的源文件都放在src文件夹里面。在src内部有res文件夹存放资源文件，library文件夹存放跨功能使用的公共文件，以及screens文件夹存放和屏幕相关的文件。

尽可能少的依赖

由于React Native严重依赖于大量的依赖项，因此在添加更多依赖项时我会非常清楚。在我的项目中，我只用react-navigation做导航。而且我不喜欢redux，因为它增加了不必要的复杂性。只有在你真正需要它时才添加一个依赖项，否则你只是为了让自己陷入更多麻烦而不是价值。

React中我喜欢的是组件。组件是我们定义视图，样式和行为的地方。React具有内联样式 - 就像使用JavaScript来定义脚本，HTML和CSS一样。这符合我们的目标特征。这就是我不使用样式组件的原因。由于样式只是JavaScript对象，我们可以在library中共享样式组件。

src

我非常喜欢Android，所以我命名src和res匹配其文件夹约定。 react-native init为我们设置了babel。但对于典型的JavaScript项目，最好在src文件夹中组织文件。在我的electron.js应用程序IconGenerator中，我将源文件放在src文件夹中。这不仅有助于组织，还有助于babel立即转换整个文件夹。只是一个命令，瞬间我将文件src转换为dist。

babel ./src --out-dir ./dist --copy-files

Screen

React基于组件，有容器和表示组件，但我们可以组成组件来构建更复杂的组件。它们通常以全屏显示结束。它在Windows Phone被称为Page，在iOS中被称为ViewController，在Android中被称为Activity。React Native文档经常提到屏幕覆盖整个空间：

移动应用很少由单个屏幕组成。管理多个屏幕的呈现和转换通常由所谓的导航器来处理。

index.js or not?

每个屏幕都被视为每个功能的入口点。您可以利用Node模块功能重命名LoginScreen.js为index.js：

模块不必是文件。我们还可以在node_modules下创建一个find-me文件夹并在其中放置index.js文件。require(‘find-me’)行将使用该文件夹下的index.js文件

因此我们可以用import LoginScreen from ‘./screens’来代替import LoginScreen from ‘./screens/LoginScreen’。

使用index.js封装并为该功能提供公共接口。这都是个人习惯。我自己更喜欢为文件准确命名，这就是LoginScreen.js的由来。

Navigator

react-navigation似乎是在React Native应用程序中处理导航的最受欢迎的选择。对于像用户引导这样的功能，可能有很多屏幕由堆栈导航管理，所以有OnboardingNavigator 。

您可以将Navigator视为对子屏幕或功能进行分组的内容。由于我们按功能分组，因此将Navigator放在功能文件夹中是合理的。它基本上看起来像这样：

import { createStackNavigator } from 'react-navigation'
import Welcome from './Welcome'
import Term from './Term'
const routeConfig = {
  Welcome: {
    screen: Welcome
  },
  Term: {
    screen: Term
  }
}
const navigatorConfig = {
  navigationOptions: {
    header: null
  }
}
export default OnboardingNavigator = createStackNavigator(routeConfig, navigatorConfig)

库

这是构建项目中最具争议的部分。如果你不喜欢这个名字library，你可以将其命名utilities，common，citadel ，whatever…

这不适用于无家可归文件，但它是我们放置常用功能和组件的地方。诸如原子组件，包装器，快速修复功能，网络内容和登录信息之类的东西被大量使用，并且很难将它们移动到特定的功能文件夹。有时我们只需要务实并完成工作。

在React Native中，我们经常需要在许多屏幕中实现带有图像背景的按钮。这是一个在里面library/components/ImageButton.js里的简单的功能。该components文件夹用于可重用组件，有时称为原子组件。根据React命名约定，第一个字母应该是大写的。

import React from 'react'
import { TouchableOpacity, View, Image, Text, StyleSheet } from 'react-native'
import images from 'res/images'
import colors from 'res/colors'
export default class ImageButton extends React.Component {
  render() {
    return (
      <TouchableOpacity style={styles.touchable} onPress={this.props.onPress}>
        <View style={styles.view}>
          <Text style={styles.text}>{this.props.title}</Text>
        </View>
        <Image
          source={images.button}
          style={styles.image} />
      </TouchableOpacity>
    )
  }
}
const styles = StyleSheet.create({
  view: {
    position: 'absolute',
    backgroundColor: 'transparent'
  },
  image: {
},
  touchable: {
    alignItems: 'center',
    justifyContent: 'center'
  },
  text: {
    color: colors.button,
    fontSize: 18,
    textAlign: 'center'
  }
})

如果我们想将按钮放在底部，我们使用utility function来防止代码重复。例如可以放在library/utils/moveToBottom.js：

import React from 'react'
import { View, StyleSheet } from 'react-native'
function moveToBottom(component) {
  return (
    <View style={styles.container}>
      {component}
    </View>
  )
}
const styles = StyleSheet.create({
  container: {
    flex: 1,
    justifyContent: 'flex-end',
    marginBottom: 36
  }
})
export default moveToBottom

使用package.json来避免相对路径

在js文件中，例如src/screens/onboarding/term/Term.js ，我们可以使用相对路径导入：

import moveToBottom from '../../../../library/utils/move'
import ImageButton from '../../../../library/components/ImageButton'

这是我眼中的一个大大的警告。这很容易出错，因为我们需要计算很多我们需要执行的..数量。如果我们移动了feature，则需要重新计算所有路径。

由于库的意图是用于许多地方，因此将其作为绝对路径引用是很好的。在JavaScript中，单个问题通常有1000个库。在Google上快速搜索可以发现大量的库来解决这个问题。但是我们不需要另外的依赖，因为这非常容易修复。

解决的办法是把library变成module，这样node可以找到它。添加package.json到任何文件夹使其成为节点module 。使用以下简单内容package.json在library文件夹中添加：

{
  "name": "library",
  "version": "0.0.1"
}

现在，在Term.js文件中我们可以很容易地导入library，因为它现在是module：

import React from 'react'
import { View, StyleSheet, Image, Text, Button } from 'react-native'
import strings from 'res/strings'
import palette from 'res/palette'
import images from 'res/images'
import ImageButton from 'library/components/ImageButton'
import moveToBottom from 'library/utils/moveToBottom'
export default class Term extends React.Component {
  render() {
    return (
      <View style={styles.container}>
        <Text style={styles.heading}>{strings.onboarding.term.heading.toUpperCase()}</Text>
        {
          moveToBottom(
            <ImageButton style={styles.button} title={strings.onboarding.term.button.toUpperCase()} />
          )
        }
      </View>
    )
  }
}
const styles = StyleSheet.create({
  container: {
    flex: 1,
    alignItems: 'center'
  },
  heading: {...palette.heading, ...{
    marginTop: 72
  }}
})

res

你可能不知道在上面的例子中的res/colors，res/strings ，res/images和res/fonts。好吧，对于前端项目，我们通常使用字体，本地化字符串，颜色，图像和样式来组件和样式。JavaScript是一种非常动态的语言，并且很容易在任何地方使用字符串类型。我们可以在许多文件中定义颜色#00B75D，或者在许多Text组件使用Fira字体作为fontFamily。这很容易出错并且难以重构。

让我们在res中用更安全的对象封装资源。它们看起来像下面的例子：

和library一样，res文件可以从任何地方访问，所以通过以下代码让它成为一个module 。

添加package.json到res文件夹：

{
  "name": "res",
  "version": "0.0.1"
}

所以我们可以像普通模块那样访问资源文件：

import strings from 'res/strings'
import palette from 'res/palette'
import images from 'res/images'

用调色板分组颜色，图像，字体

应用程序的设计应该是一致的。某些元素应该具有相同的外观和感觉，因此它们不会混淆用户。例如，标题Text应使用一种颜色，字体和字体大小。Image组件应使用相同的占位符图像。在React Native中，我们已经使用了名称styles，const styles = StyleSheet.create({}) 所以让我们使用名称palette。

下面是我简单的调色板。它定义了标题的常用样式和Text：

res/palette
import colors from './colors'
const palette = {
  heading: {
    color: colors.title,
    fontSize: 20,
    textAlign: 'center'
  },
  text: {
    color: colors.text,
    fontSize: 17,
    textAlign: 'center'
  }
}
export default palette

然后我们可以在屏幕上使用它们：

const styles = StyleSheet.create({
  container: {
    flex: 1,
    alignItems: 'center'
  },
  heading: {...palette.heading, ...{
    marginTop: 72
  }}
})

这里我们使用对象扩展运算符来合并palette.heading和我们的自定义样式对象。这意味着我们使用了palette.heading中的样式，但也指定了更多属性。

如果我们要为多个品牌重新设置应用程序，我们可能会有多个调色板。这是一个非常强大的模式。

生成图像

你可以看到在/src/res/images.js中的每个图像都有属性：

const images = {
  button: require('./images/button.png'),
  logo: require('./images/logo.png'),
  placeholder: require('./images/placeholder.png')
}
export default images

这是手动执行的繁琐工作，如果图像命名约定发生变化，我们必须更新。相反，我们可以添加一个脚本来根据我们的图像生成images.js。 在项目/scripts/images.js的根目录添加一个文件：

const fs = require('fs')
const imageFileNames = () => {
  const array = fs
    .readdirSync('src/res/images')
    .filter((file) => {
      return file.endsWith('.png')
    })
    .map((file) => {
      return file.replace('@2x.png', '').replace('@3x.png', '')
    })
return Array.from(new Set(array))
}
const generate = () => {
  let properties = imageFileNames()
    .map((name) => {
      return `${name}: require('./images/${name}.png')`
    })
    .join(',\n  ')
const string = `const images = {
  ${properties}
}
export default images
`
fs.writeFileSync('src/res/images.js', string, 'utf8')
}
generate()

关于Node的一个很酷的事情是我们可以访问fs模块，它非常擅长文件处理。在这里，我们只是遍历图像，并相应地更新/src/res/images.js。

每当我们添加或更改图像时，我们都可以运行：

node scripts/images.js

我们也可以在main中声明脚本package.json ：

"scripts": {
  "start": "node node_modules/react-native/local-cli/cli.js start",
  "test": "jest",
  "lint": "eslint *.js **/*.js",
  "images": "node scripts/images.js"
}

现在我们可以运行npm run images并获得最新的images.js 资源文件。

自定义字体怎么样？

React Native有一些自定义字体可能对你的项目来说足够好。您也可以使用自定义字体。

需要注意的一点是，Android使用字体文件的名称，但iOS使用全名。您可以在Font Book应用程序中查看全名，也可以在正在运行的应用程序中查看

for (NSString* family in [UIFont familyNames]) {
  NSLog(@"%@", family);
for (NSString* name in [UIFont fontNamesForFamilyName: family]) {
    NSLog(@"Family name:  %@", name);
  }
}

对于要在iOS的注册自定义字体，我们需要声明UIAppFonts中Info.plist使用的字体的文件名，在Android开发中，需要的字体被放置在app/src/main/assets/fonts。

最好将字体文件命名为全名。据说React Native会动态加载自定义字体，但是如果你得到“无法识别的字体系列”，那么只需将这些字体添加到Xcode中的目标。

手工完成这项工作需要时间，幸运的是我们有rnpm可以提供帮助。首先添加res/fonts文件夹中的所有字体。然后简单地声明rnpm中package.json并运行react-native link 。这应该在iOS中声明UIAppFonts，在Android中将所有字体移动到app/src/main/assets/fonts。

"rnpm": {
  "assets": [
    "./src/res/fonts/"
  ]
}

按名称访问字体容易出错，我们可以创建一个类似于我们对图像所做的脚本来生成更安全的加入。添加fonts.js到我们的scripts文件夹

const fs = require('fs')
const fontFileNames = () => {
  const array = fs
    .readdirSync('src/res/fonts')
    .map((file) => {
      return file.replace('.ttf', '')
    })
return Array.from(new Set(array))
}
const generate = () => {
  const properties = fontFileNames()
    .map((name) => {
      const key = name.replace(/\s/g, '')
      return `${key}: '${name}'`
    })
    .join(',\n  ')
const string = `const fonts = {
  ${properties}
}
export default fonts

fs.writeFileSync('src/res/fonts.js', string, 'utf8')
}

generate()

import R from 'res/R'

  text: {
    fontFamily: R.fonts.FireCodeNormal

R 命名空间

这一步取决于个人品味，但如果我们引入R命名空间，我发现它更有条理，就像Android对生成的R类的assets所做的那样。

外部化应用程序资源后，可以使用项目R类中生成的资源ID访问它们。本文档介绍如何在Android项目中对资源进行分组，并为特定设备配置提供备用资源，然后从应用程序代码或其他XML文件中访问它们。

通过这种方式，让我们在src/res中生成一个名为R.js的文件：

import strings from './strings'
import images from './images'
import colors from './colors'
import palette from './palette'
const R = {
  strings,
  images,
  colors,
  palette
}
export default R

并在屏幕上访问它：

import R from 'res/R'
render() {
  return (
    <SafeAreaView style={styles.container}>
      <Image
        style={styles.logo}
        source={R.images.logo} />
      <Image
        style={styles.image}
        source={R.images.placeholder} />
      <Text style={styles.title}>{R.strings.onboarding.welcome.title.toUpperCase()}</Text>
  )
}

strings替换为R.strings，colors 替换为R.colors，images替换为R.images。使用R注释，很明显我们正在从应用程序包中访问静态资产。
